# Karpathy Wiki — Design Spec

A markdown-based personal knowledge base CLI built on Karpathy's LLM Wiki pattern, with NVIDIA NIM embeddings, agent skill integration, and tailnet-friendly sync.

## 1. Architecture overview

### Stack

- **Wiki storage:** Markdown files in `wiki/`, sources in `raw/`, embeddings in `embeddings.jsonl` (gitignored). No database anywhere.
- **CLI:** Rust crate `wiki` (single static binary). Uses `clap`, `reqwest`, `tokio`, `notify`, `serde_yaml`, `walkdir`, `pulldown-cmark`, `anyhow`/`thiserror`, `tracing`.
- **Viewer:** SvelteKit + `adapter-static`. Builds static HTML/JS that consumes `wiki/*.md` + `embeddings.jsonl` at build time. No live API server.
- **Skill:** 15-line `SKILL.md` stub (generated by `build.rs`) + full content embedded in the binary via `include_str!`, served by `wiki skill show [topic]`.
- **Embeddings:** NVIDIA NIM (`nvidia/nv-embed-v1` default, 4096 dims). Configurable to any of 8 supported NIM models. JSONL append-only log.
- **Sync:** Wiki folder is a git repo. Markdown diffs are the only thing on the wire between devices. Embeddings regenerate on demand per device.

### Cross-device view (tailnet, git-synced)

```
Device A (host)  ─git push─►  remote (GitHub/Tailscale git)  ─git pull─►  Device B
                                                                       
   wiki/  ◄───────────── markdown only (canonical) ─────────────►  wiki/
   embeddings.jsonl         (regenerated per device)              embeddings.jsonl
```

### Design principles

1. **Filesystem is the only source of truth.** Delete `embeddings.jsonl` → only speed is lost, no data.
2. **Agent-first.** Skill teaches agent to use its own tools first; CLI is for semantic search only.
3. **No background daemons required.** `wiki watch` is opt-in.
4. **Cross-device parity via git, not network mounts.** Each device is independent.

### Simplifications vs MyWiki

- **No mandatory `concepts/` `entities/` `comparisons/` `queries/` split.** The CLI treats `wiki/` as one flat namespace.
- **Lighter frontmatter:** `title`, `created`, `updated`, `tags`, `sources` required. `type`, `confidence`, `contested`, `review_by` are optional.
- **Filenames:** lowercase-hyphen regex enforced for new pages.
- **No AGENTS.md in the user's wiki.** The skill carries that role globally.
- **AGENTS.md at the tool repo root** for dev-related core knowledge about the tool itself.

## 3. CLI command surface

Single Rust binary `wiki`. Every command supports `--json` for headless output. Every command supports `--workspace <path>` to override workspace discovery.


### Global flags

- `--workspace <path>` — override workspace discovery
- `--config <path>` — use a specific config file
- `--no-color` — disable ANSI
- `--json` — machine-readable output
- `--verbose` — increase log verbosity

### Output conventions

- Default: human-friendly, color when TTY
- `--json`: single JSON object on stdout, logs to stderr
- Exit codes: `0` success, `1` user error, `2` lint errors found, `3` NIM unreachable, `4` config invalid

### Workspace discovery (priority order)

1. `--workspace <path>` flag
2. `WIKI_WORKSPACE` env var
3. Walk up from CWD looking for `.wiki/` directory
4. `~/wiki` if it exists

### Config resolution (cascade, later wins)

1. Built-in defaults (in code)
2. `~/.config/wiki/config.yaml` (global)
3. `<wiki>/.wiki/config.yaml` (workspace)
4. `--config` flag

- The `wiki` binary is the tool — lives in the repo at `/Users/felix/Documents/llmwiki/`.

### Rust implementation

**`src/skills/mod.rs`:**

```rust
pub const SKILL_MD: &str = include_str!("skill_md.md");

pub const TOPIC_SETUP: &str = include_str!("topics/setup.md");
pub const TOPIC_INGEST: &str = include_str!("topics/ingest.md");
pub const TOPIC_SEARCH: &str = include_str!("topics/search.md");
pub const TOPIC_QUERY: &str = include_str!("topics/query.md");
pub const TOPIC_LINT: &str = include_str!("topics/lint.md");
pub const TOPIC_MODELS: &str = include_str!("topics/models.md");
pub const TOPIC_SYNC: &str = include_str!("topics/sync.md");
pub const TOPIC_TROUBLESHOOTING: &str = include_str!("topics/troubleshooting.md");

pub const TOPICS: &[(&str, &str)] = &[
    ("setup", TOPIC_SETUP),
    ("ingest", TOPIC_INGEST),
    ("search", TOPIC_SEARCH),
    ("query", TOPIC_QUERY),
    ("lint", TOPIC_LINT),
    ("models", TOPIC_MODELS),
    ("sync", TOPIC_SYNC),
    ("troubleshooting", TOPIC_TROUBLESHOOTING),
];

pub fn find_topic(name: &str) -> Option<&'static str> {
    TOPICS.iter()
        .find(|(k, _)| k.eq_ignore_ascii_case(name))
        .map(|(_, v)| *v)
}

pub fn list_topics() -> Vec<(&'static str, usize)> {
    TOPICS.iter().map(|(k, v)| (*k, v.lines().count())).collect()
}
```

**`src/commands/skill.rs`:**

```rust
use clap::{Args, Subcommand};

#[derive(Args)]
pub struct SkillArgs {
    #[command(subcommand)]
    pub cmd: Option<SkillCmd>,
}

#[derive(Subcommand)]
pub enum SkillCmd {
    /// Print the full SKILL.md content
    Show {
        topic: Option<String>,
    },
    /// List available skill topics
    List,
}

pub fn run(args: SkillArgs) -> anyhow::Result<()> {
    match args.cmd {
        None | Some(SkillCmd::Show { topic: None }) => {
            print!("{}", crate::skills::SKILL_MD);
        }
        Some(SkillCmd::Show { topic: Some(t) }) => {
            match crate::skills::find_topic(&t) {
                Some(content) => print!("{}", content),
                None => {
                    eprintln!("Unknown topic '{}'. Available:", t);
                    for (name, lines) in crate::skills::list_topics() {
                        eprintln!("  {:20} ({} lines)", name, lines);
                    }
                    std::process::exit(1);
                }
            }
        }
        Some(SkillCmd::List) => {
            for (name, lines) in crate::skills::list_topics() {
                println!("{:20} ({} lines)", name, lines);
            }
        }
    }
    Ok(())
}
```

### Distribution model

**Option A — `cargo install --path .`** (the user's normal flow):
1. Build runs `build.rs`, which generates `agents/skills/wiki/SKILL.md` in the repo
2. `cargo install` copies the binary to `~/.cargo/bin/wiki`
3. Skill content is baked into the binary via `include_str!`
4. The user runs `wiki install-skill --global` once to symlink the stub
5. The agent reads the stub, sees the `wiki skill show` pointer, and calls it on demand

**Option B — Kimi Code plugin** (parallel distribution):
1. Add `kimi.plugin.json` to the tool repo
2. Kimi Code auto-discovers `wiki` as a built-in-tier skill
3. Same stub, same content

Both A and B coexist. Same stub, same content. The binary is the source of truth.

### Progressive disclosure benefit

- **Discovery (always loaded):** 15-line stub with `description:` for auto-matching
- **On first use of `wiki`:** Agent calls `wiki skill show` → gets the full ~250-line guide, ~10KB of context
- **For specific tasks:** Agent calls `wiki skill show search` → gets only the ~30-line search topic, ~1KB

Net context reduction vs the user's existing 887-line wiki skill: ~95% in the typical case.

## 5. Embedding pipeline

The pipeline turns wiki markdown → NIM vectors → `embeddings.jsonl` on disk.

### Storage format — `embeddings.jsonl`

One JSON object per line, append-only:

```jsonl
{"path": "wiki/concepts/attention.md", "sha256": "9f86d081...", "model": "nvidia/nv-embed-v1", "dim": 4096, "chunked": true, "chunks": [{"start": 0, "end": 1820, "tokens": 412, "embedding": [0.0123, -0.0456, ...]}, {"start": 1500, "end": 3400, "tokens": 421, "embedding": [0.0789, ...]}], "embedded_at": "2026-06-21T10:30:00Z"}
```

**Field semantics:**
- `path` — relative to workspace root
- `sha256` — content hash; lets `wiki embed --skip-existing` skip unchanged pages
- `model` — exact model used (so multiple models can coexist in the same JSONL)
- `dim` — embedding dimension (4096 for NV-Embed v1; can be lower if Matryoshka-sliced)
- `chunked` — false if whole page fit in one chunk
- `chunks[].start/end` — byte offsets into the file
- `chunks[].embedding` — float array
- `embedded_at` — ISO-8601 timestamp

### Pipeline stages

```
┌────────────────────┐    ┌─────────────────────┐    ┌──────────────────────┐
│   wiki embed       │ ─► │  Chunk markdown     │ ─► │  POST to NIM         │
│   (CLI command)    │    │  (chunker.rs)       │    │  /v1/embeddings      │
└────────────────────┘    └─────────────────────┘    └─────────┬────────────┘
                                                              │
                                                              ▼
                                                  ┌──────────────────────┐
                                                  │  Append to           │
                                                  │  embeddings.jsonl    │
                                                  └──────────────────────┘
```

### Chunker

Ported from llmwiki `mcp/services/chunker.py`:
- ~512 token chunks, ~128 token overlap (configurable)
- Splits on paragraph boundaries (`\n\n`) first, then sentences for oversized chunks
- Records `header_breadcrumb` per chunk for context
- Skips chunks below `MIN_CHUNK_TOKENS` (32)

### NIM HTTP client

- Endpoint: `POST {base_url}/v1/embeddings`
- Body: `{"input": ["text1", "text2", ...], "model": "nvidia/nv-embed-v1", "input_type": "passage"}`
- For queries: `"input_type": "query"`
- `reqwest` + `tokio` for async batching
- `--batch-size 8` configurable
- 30s timeout, 3 retries with exponential backoff (500ms base)

### `wiki embed` behavior

| Flag | Effect |
|------|--------|
| (none) | Full re-embed of all `wiki/**/*.md` |
| `--skip-existing` | Read JSONL; skip pages where `sha256` matches an existing entry with the same `model` |
| `--model <name>` | Override configured model; new entries tagged with new model |
| `--dims <n>` | Matryoshka-slice the embedding to first N dimensions before writing |
| `--watch` | Daemon: re-embed on file change |
| `--batch-size <n>` | Parallel NIM requests |

### `wiki embed --watch` daemon

- Uses `notify::RecommendedWatcher` on the `wiki/` directory
- On `Modify` or `Create` for `.md` files: re-embed that single page
- On `Remove`: remove the page's entries from `embeddings.jsonl`
- Debounces writes (300ms)
- Writes a PID file at `<workspace>/.wiki/embed-watch.pid` for clean termination
- Logs to `<workspace>/.wiki/embed-watch.log`

### NV-Embed v1 specifics

- Embedding dim: 4096 native
- Max context: 32k tokens (we chunk to 512 anyway)
- `input_type`: `query` for search/query; `passage` for indexed chunks
- License: **non-commercial only** — surfaced as a warning in `wiki models` and `wiki doctor`

### JSONL hygiene

- File is rewritten atomically: write to `embeddings.jsonl.tmp`, fsync, rename
- Removed pages are pruned in the same pass
- `wiki doctor` verifies: every path in `wiki/` has an entry; every entry in JSONL still exists on disk

### Performance budget

- 1000 pages × 4096 dims × 4 bytes/float ≈ 16 MB resident — trivially fast
- Cosine similarity = dot product since vectors are normalized (NIM returns L2-normalized)
- Search over 1000 chunks on a modern laptop: <100ms in pure Rust

## 6. Search & query

Two commands: **semantic search** (returns ranked chunks) and **RAG query** (returns synthesized answer with citations).

### `wiki search <query>`

```
┌──────────────────┐    ┌───────────────────┐    ┌──────────────────────┐
│ Embed query      │ ─► │ Cosine similarity │ ─► │ Optional rerank      │
│ (NIM, query)     │    │ vs all chunks     │    │ (NIM rerank, if on)  │
└──────────────────┘    └───────────────────┘    └──────────┬───────────┘
                                                           │
                                                           ▼
                                              ┌──────────────────────────┐
                                              │ Return top-K matches     │
                                              │ with file path + score   │
                                              └──────────────────────────┘
```

**Step 1 — Embed the query**
- Single POST to `{base_url}/v1/embeddings` with `input=[query]`, `input_type="query"`
- Returns a single 4096-dim vector

**Step 2 — Cosine similarity over all chunks**
- Load `embeddings.jsonl` (L2-normalized)
- For each chunk: `score = dot(query_vec, chunk_vec)`
- Sort descending, take top-50 candidates
- Apply `--threshold <f>` filter (default 0.3)
- If `--rerank` enabled: send top-50 to NIM reranker, reorder, take top-K
- Otherwise: take top-K directly (default 10)

**Step 3 — Format output**

Default (human):
```
$ wiki search "sparse attention mechanisms"

✓ 3 result(s) for "sparse attention mechanisms":

  [0.847] wiki/concepts/attention.md  (chunk 2, lines 89-142)
    "...Sparse attention reduces the quadratic cost of self-attention
    by restricting each token to attend to only a subset..."

  [0.812] wiki/concepts/transformers.md  (chunk 5, lines 230-289)
    "...Variants like Longformer, BigBird, and Routing Transformers
    implement sparse patterns..."

  [0.751] wiki/concepts/efficient-inference.md  (chunk 0, lines 1-65)
    "...Sparse attention is one of several efficiency techniques..."

Run `wiki query "..."` for a synthesized answer with citations.
```

JSON (`--json`):
```json
{
  "query": "sparse attention mechanisms",
  "model": "nvidia/nv-embed-v1",
  "results": [
    {"path": "wiki/concepts/attention.md", "chunk_index": 2, "start": 4521, "end": 7102, "score": 0.847, "snippet": "..."}
  ]
}
```

### `wiki query <question>` — RAG pipeline

```
┌──────────────────┐    ┌──────────────────┐    ┌────────────────────┐
│ Embed question   │ ─► │ Retrieve top-K   │ ─► │ Build prompt with  │
│ (NIM, query)     │    │ chunks           │    │ chunks as context  │
└──────────────────┘    └──────────────────┘    └─────────┬──────────┘
                                                           │
                                                           ▼
                                              ┌────────────────────────┐
                                              │ POST to NIM LLM        │
                                              └─────────┬──────────────┘
                                                        │
                                                        ▼
                                              ┌────────────────────────┐
                                              │ Return answer +        │
                                              │ citation list          │
                                              └────────────────────────┘
```

- Default top-K = 5
- System prompt: "Use ONLY the provided context. Cite with [^N]."
- Default LLM: `meta/llama-3.1-70b-instruct`
- 1024 max tokens, temperature 0.2
- Citations mapped to context chunk paths

### Reranker

- Opt-in via `nim.rerank_model` in config or `--rerank` flag
- Takes top-20 from embedding search, sends to NIM rerank endpoint
- Returns reordered top-K
- Adds ~200-400ms latency per query
- Default reranker if enabled: `nvidia/llama-nemotron-rerank-1b-v2`

### Edge cases

| Case | Behavior |
|------|----------|
| No embeddings yet | Error: "Run `wiki embed` first" |
| Empty wiki | Error: "No pages indexed yet" |
| All below threshold | Empty results + hint: "Try `wiki search` with broader terms" |
| NIM LLM unreachable | Error + fallback to `wiki search` results |
| Many pages in results | Deduplicate by page |

### Performance budget

- `wiki search`: <500ms for 1000 pages
- `wiki query`: 2-5s

## 7. Lint rules

`wiki lint` runs deterministic checks. Mirrors llmwiki's `mcp/tools/lint.py` adapted to the simpler frontmatter schema.

**Exit codes:**
- `0` — no issues
- `2` — errors found (blocking)
- `1` — internal lint failure

### Frontmatter checks

| Code | Severity | Description |
|------|----------|-------------|
| `missing-frontmatter` | error | No `---` block |
| `missing-title` | error | No `title` |
| `missing-created` | error | No `created` |
| `missing-updated` | error | No `updated` |
| `missing-tags` | error | No `tags` |
| `missing-sources` | error | No `sources` (or empty array) |
| `bad-filename` | error | Filename doesn't match regex |
| `invalid-date` | warn | Date not YYYY-MM-DD |
| `updated-before-created` | warn | `updated` < `created` |
| `old-page` | warn | `updated` > 6 months ago |
| `duplicate-tags` | warn | Same tag listed twice |
| `unknown-type` | warn | `type` not in {entity, concept, comparison, query, summary} |

### Wikilink checks

| Code | Severity | Description |
|------|----------|-------------|
| `dangling-link` | error | `[[page-name]]` doesn't resolve |
| `dead-end` | warn | 0 outbound `[[wikilinks]]` |
| `orphan` | warn | 0 inbound `[[wikilinks]]` |
| `few-outbound-links` | warn | < 2 outbound links |
| `link-to-self` | warn | Page links to itself |
| `case-mismatch` | warn | Case-mismatched link target |

### Footnote & citation checks

| Code | Severity | Description |
|------|----------|-------------|
| `footnote-undefined` | error | `[^N]` used but no `[^N]:` definition |
| `footnote-unused` | warn | `[^N]:` defined but never referenced |
| `footnote-duplicate` | error | Same `[^N]` defined twice |
| `citation-unresolved` | error | `[^N]:` points to file not in `raw/` |
| `source-not-cited` | warn | File in `raw/` not cited |
| `mid-document-footnotes` | warn | Footnote definitions not grouped at end |

### Index.md consistency checks

| Code | Severity | Description |
|------|----------|-------------|
| `page-not-in-index` | error | Page in `wiki/` not in `index.md` |
| `index-points-to-missing` | error | `index.md` lists missing page |
| `page-in-index-multiple-times` | error | Same path appears twice |
| `index-no-section` | warn | Page not in any `## Section` |

### Raw/ immutability checks

| Code | Severity | Description |
|------|----------|-------------|
| `raw-no-frontmatter` | warn | Non-binary file lacks frontmatter |
| `raw-no-sha256` | warn | Raw file lacks `sha256` |
| `raw-drift` | error | Recomputed `sha256` doesn't match declared |
| `raw-orphan` | warn | Raw file not referenced by any page |

### Embeddings.jsonl consistency checks

| Code | Severity | Description |
|------|----------|-------------|
| `embeddings-stale` | warn | Page `sha256` differs from JSONL |
| `embeddings-orphan` | warn | JSONL path no longer exists |
| `embeddings-empty` | warn | `wiki/` has pages but JSONL is empty |
| `embeddings-model-mismatch` | warn | Some entries use different model |

### Log.md hygiene checks

| Code | Severity | Description |
|------|----------|-------------|
| `log-missing-entry` | warn | Page modified in last 7 days with no log entry |
| `log-bad-format` | warn | Entry doesn't match format |

### Flags

- `--scope wiki|raw|all` (default: all)
- `--strict` (warnings become errors)
- `--fix` (auto-fix safe issues)
- `--json`
- `--quiet` (summary only)

### Auto-fix scope

Safe to fix automatically:
- Add missing entries to `index.md`
- Add missing `created` or `updated` dates (using file mtime)
- Lowercase filenames (with git mv)
- Add empty frontmatter block if missing

NOT auto-fixed: dangling wikilinks, citation failures, orphan/dead-end pages, raw frontmatter drift.

## 8. Tailnet / multi-device sync

The wiki is a git repo. Markdown diffs are the only thing on the wire. Embeddings regenerate locally per device.

### Per-device state

**Shared (git-tracked):**
- `wiki/` — compiled markdown
- `raw/` — source materials
- `index.md` — page catalog
- `log.md` — operational log
- `.wiki/config.yaml` — workspace config
- `.gitignore`
- `.agents/skills/wiki/` — skill stub (if installed per-workspace)

**Per-device (gitignored):**
- `embeddings.jsonl`
- `.wiki/embed-watch.pid`
- `.wiki/embed-watch.log`
- `.wiki/cache/`

### Workflow

```bash
# Pull latest
cd ~/my-wiki && git pull

# First search on this device since last embed → re-embed
wiki search "..."  # auto-detects: if embeddings.jsonl stale, run `wiki embed --skip-existing`

# Or manually
wiki embed --skip-existing

# Edit / add pages (agent does this via Read/Write/Edit tools)

# Commit & push
git add . && git commit -m "ingest: <source>" && git push
```

### Conflict handling

- **Markdown conflicts:** Git's normal merge (rare for typical wiki edits).
- **Embeddings.jsonl conflicts:** Always gitignored.
- **log.md conflicts:** Append-only pattern; resolve by keeping both blocks + post-merge lint pass.
- **raw/ conflicts:** Raw files are immutable; new content goes in new files.

### Tailnet access patterns

- **Pattern 1:** One device is the wiki host. `wiki serve --bind 0.0.0.0 --port 4173` exposes viewer to tailnet.
- **Pattern 2:** Each device has its own checkout. Embeddings always local.
- **Pattern 3:** Read-only consumer (VPS) clones repo, serves behind reverse proxy (Tailscale Serve, Caddy, nginx).

### Performance across devices

- Cold start on new device: `git clone` + first `wiki embed` (~5-10min for 1000 pages)
- Subsequent: `git pull` + `wiki embed --skip-existing` (only changed pages)

### What is NOT in v1

- No automatic `git pull` daemon
- No automatic push on wiki edit
- No merge conflict UI
- No cross-device lock

## 9. Svelte static viewer

SvelteKit app with `adapter-static`. Reads wiki (markdown + `embeddings.jsonl`) at build time, produces static site. No live API, no Node server, no auth surface.

### Build pipeline

```
┌────────────────────────┐    ┌─────────────────────┐    ┌──────────────────────┐
│ wiki serve             │ ─► │ svelte-kit sync     │ ─► │ vite build           │
│ (rebuilds if needed)   │    │ (copies wiki/ +     │    │ (static export to    │
│                        │    │  embeddings.jsonl   │    │  web/build/)         │
│                        │    │  into src/lib/wiki) │    │                      │
└────────────────────────┘    └─────────────────────┘    └──────────┬───────────┘
                                                                   │
                                                                   ▼
                                                      ┌──────────────────────────┐
                                                      │ python -m http.server    │
                                                      │ (or wiki serve)          │
                                                      │ on tailnet IP:4173       │
                                                      └──────────────────────────┘
```

### Routes

| Route | Purpose |
|-------|---------|
| `/` | Wiki home — list of pages |
| `/page/[...slug]` | Single wiki page with backlinks sidebar |
| `/search` | Semantic search UI (client-side cosine sim) |
| `/graph` | Force-directed graph of `[[wikilinks]]` |
| `/raw/[...path]` | View raw source files |

### Page rendering

- `marked` or `markdown-it` in browser
- Custom renderer for `[[wikilinks]]` → `<a href="/page/{slug}">`
- Custom renderer for `[^N]` footnote references
- Frontmatter rendered as header card

### Search UI (client-side)

- Loads `embeddings.jsonl` lazily on first search
- Computes query embedding via fetch to NIM endpoint
- Ranks chunks by cosine similarity
- Shows inline snippet + path

### Build-time data loading

Reads:
1. `wiki/**/*.md` — parsed for frontmatter, content, wikilinks
2. `embeddings.jsonl` — indexed by path for search
3. `index.md` — page catalog
4. `raw/**/*.{md,txt}` — available for `/raw/[...path]`

Generated:
- `src/lib/wiki/pages.json` — `{slug, title, tags, sources, updated, content_html, backlinks[]}`
- `src/lib/wiki/embeddings.json` — flattened array
- `src/lib/wiki/graph.json` — `{nodes: [...], edges: [...]}`

### Performance

- `embeddings.jsonl` size: ~80MB worst case (1000 pages × 4096 dims × 4 bytes × ~5 chunks). Lazy-load only on `/search`.
- Up to 5000 pages at build time is fine.
- Build time: ~30s for 500 pages.

## 10. Configuration & secrets

### Layered config (cascade, later wins)

```
Priority (highest to lowest):
1. CLI flags
2. Environment variables
3. Workspace config (~/my-wiki/.wiki/config.yaml)
4. Global config (~/.config/wiki/config.yaml)
5. Built-in defaults
```

### Env vars

| Env var | Purpose |
|---------|---------|
| `NVIDIA_NIM_API_KEY` | Required: NIM API auth |
| `WIKI_WORKSPACE` | Override workspace discovery |
| `WIKI_CONFIG` | Override config file path |
| `WIKI_NO_COLOR` | Disable color |
| `WIKI_LOG_LEVEL` | `trace`, `debug`, `info`, `warn`, `error` |
| `WIKI_NIM_BASE_URL` | Override NIM base URL |
| `WIKI_EMBED_MODEL` | Override embed model |
| `NO_COLOR` | Standard: disable color |

### Secrets handling

1. **Never write secrets to YAML.** Config contains `api_key_env: "NVIDIA_NIM_API_KEY"`, not the key.
2. **Never log secrets.** `tracing` redacts `*_key`, `*_token`, `*_secret`, `password`, `api_key`.
3. **Never commit secrets.** `.gitignore` includes `.env`, `.env.local`, `*.key`, `secrets.yaml`.
4. **Never include secrets in `--json` output.** Doctor/status return `set` / `unset` only.
5. **NIM API key validation:** `wiki doctor` validates by hitting `/v1/models` HEAD, reports validity without revealing.

### Config validation

On startup:
1. Load all config layers (deep merge)
2. Validate against JSON schema
3. Report errors with line numbers
4. Exit code 4 if invalid

### Schema versioning

```yaml
config_version: 1
```

CLI handles migrations automatically.

### Config documentation

`docs/config.md` documents every field with type, default, layer, example, pitfalls. Auto-generated from Rust struct doc comments via `build.rs`.

## Appendix B: Decision log

Key decisions made during brainstorming (2026-06-21):

1. **Gut llmwiki in place** vs layer on top vs fresh repo: chose gut-in-place (single coherent project, share llmwiki conventions).
2. **MVP scope:** Full stack (markdown + NIM + watcher + multi-workspace + viewer).
3. **Tailnet access:** Git-synced workspaces (each device independent, embeddings regenerated locally).
4. **CLI binary name:** Fresh `wiki` binary (clearer than extending gutted llmwiki).
5. **Embedding storage:** JSONL index + gitignore (regenerated per device).
6. **NIM model default:** `nvidia/nv-embed-v1` (best quality, non-commercial warning surfaced).
7. **Architecture:** Approach 2 — Lean + static viewer.
8. **Viewer stack:** Svelte (lighter than Next.js).
9. **CLI language:** Rust (single static binary, fast startup, no venv).
10. **Skill pattern:** 5-line discovery stub + `include_str!` baked content + `wiki skill show` (matches `agent-browser` precedent).
