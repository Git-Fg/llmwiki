### Phase C — Branch Consolidation + Tag

```
git checkout main
git merge --no-ff feat/v0.3.0-mcp -m "Release v0.3.0: marketplace install + LSP/MCP"
git tag -a v0.3.0 -m "v0.3.0 — marketplace install + LSP/MCP sub-skill bundles"
git push origin main v0.3.0
```

The `git push ... v0.3.0` tag push will trigger `.github/workflows/release.yml` which:
- builds 6 targets (linux-musl ×2, windows-gnu ×2, apple-darwin ×2)
- publishes GitHub Release with all 6 artifacts + install.sh + install.ps1
- publish-crates.yml publishes to crates.io

**Critical:** Verify the release.yml workflow file itself before pushing the tag — it has not been exercised end-to-end yet. If it fails on first run, the tag can be moved with `git tag -d v0.3.0 && git push origin :refs/tags/v0.3.0 && git tag -a v0.3.0 <commit>`.

## Risks

| Risk                                          | Mitigation                                                              |
| --------------------------------------------- | ----------------------------------------------------------------------- |
| release.yml fails on first tag push           | Tag is movable; amend + repush. Fix forward, never rewrite tag history. |
| install.sh curl under GitHub API rate limits  | Use `/latest` redirect + `Location` header; fall back to `redirect_url` extraction. |
| install.ps1 unexercised                       | Have user run it manually on Windows before tagging (out of scope here). |
| Branch divergent from main with `--no-ff`     | Resolve any drift before merging. `git log main..feat/v0.3.0-mcp --oneline` should show only Phase A commits plus the prior feature-branch commits; nothing more. |
