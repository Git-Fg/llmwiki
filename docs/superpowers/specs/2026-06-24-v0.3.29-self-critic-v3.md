## H2 — RESOLVED with evidence

**Problem:** v0.3.29 installed to `~/.agents/skills/wiki/` which is the cross-host *generic fallback* read by [Kimi Code](https://moonshotai.github.io/kimi-cli/en/customization/skills.html) but NOT by [Claude Code](https://code.claude.com/docs/en/skills) (which reads `~/.claude/skills/`), Cursor (`.cursorrules`, not SKILL.md), or [Gemini CLI](https://medium.com/google-cloud/confused-about-where-to-put-your-agent-skills-ea778f3c64f3) (which reads `~/.gemini/skills/`).

**Fix:** added `llmwiki-cli install-skill --install-path <dir>`. The flag is **opt-in** (default behavior unchanged) so existing Kimi users see no difference; Claude Code / Gemini / Antigravity users can now point the install at their agent's skills directory.

**Test evidence (2 new tests in `tests/install_skill_test.rs`):**
- `install_skill_install_path_overrides_default`: verifies `--install-path <custom>` writes the hub to `<custom>/SKILL.md` with byte-identical content vs `llmwiki_cli::skills::hub()`.
- `install_skill_install_path_expands_tilde`: verifies `--install-path ~/foo` resolves to `$HOME/foo` when HOME is overridden.

**Manual verification:**
```bash
$ llmwiki-cli install-skill --install-path /tmp/xyz/custom-wiki
✓ Installed wiki skill to /tmp/xyz/custom-wiki
  Inline sub-skills served via `llmwiki-cli skill get <topic>`
$ diff -q skills/SKILL.md /tmp/xyz/custom-wiki/SKILL.md && echo OK
OK
```

**Limitation acknowledged in CHANGELOG:** Cursor does not yet support SKILL.md (it uses `.cursorrules`). The `--install-path` flag won't help Cursor users until Cursor adds SKILL.md support. The cross-host pattern is "wait for the host to adopt the standard" — not something we can fix from the CLI side.

## M2 — RESOLVED with explanation

**Problem:** `build.rs` listed `cargo:rerun-if-changed=skills/SKILL.md` manually, but `rust-embed` emits its own per-file rerun triggers for every file in the embedded folder. The manual entry was redundant.

**Fix:** dropped the redundant line; added a comment explaining the delegation to rust-embed:
```rust
// Note: `rust-embed` (in `src/skills/mod.rs`) emits its own
// `cargo:rerun-if-changed=` lines for every file in the `skills/`
// folder, so we don't need to list them here.
```

**Test evidence:** build still works correctly; test suite passes; cargo build cache invalidation works as expected.

## H1 — still open, recommend defer

**The remaining H1 finding (10 top-level dirs vs 1 hub) is a design tension, not a bug.** It surfaces a real cross-host advantage (external scanners like `npx skills add` would see 10 entries instead of 1) but the user's explicit "small install surface" preference argues against it. agent-browser's source confirms their pattern is the same as ours (1 stub + 9 served on demand), with the difference that they also have `hidden: true` on stubs so external scanners can find them while CLI list doesn't show them.

**Recommendation:** **ship v0.3.30 as-is** (H2 resolved); revisit H1 in v0.3.31+ only if user feedback shows external scanner visibility is a real friction point. The `--install-path` flag is the right immediate fix; the directory layout is a v0.3.31+ concern.

12. Rust-embed matches the gold-standard pattern (agent-browser uses rust-embed=8, we use 6; both v6 and v8 have the same API for our use case)

## What v0.3.30 still has wrong (cumulative, post v3)

1. **H1 still open:** discovery layout. External scanners (`npx skills add`, etc.) see only 1 entry (`wiki`) instead of 9. The fix is non-trivial (split `skills/` into `skills/` stubs + `skill-data/` content per agent-browser pattern) and the user hasn't signaled this is a problem. Defer to v0.3.31+.

2. **Cursor is fundamentally incompatible** (uses `.cursorrules`, not SKILL.md). Not something the CLI can fix. Document for users: "if you're on Cursor, see the .cursorrules equivalent at our docs."

3. **H2 partially resolved:** `--install-path` covers Claude Code + Gemini + Antigravity (which read brand paths). It does NOT help users on the Claude API (Skills are uploaded, not filesystem-based) — those users need to upload the skill via Anthropic's Skills API separately. Out of scope for our CLI.

## What agent-browser does that we don't (v0.3.31+ candidates)

From the [agent-browser source](https://github.com/vercel-labs/agent-browser/blob/main/cli/src/skills.rs):

| Feature | agent-browser has | llmwiki-cli v0.3.30 | Recommendation |
|---|---|---|---|
| `rust-embed` for skills | ✅ (`rust-embed = "8"`) | ✅ (`rust-embed = "6"`) | Done; could upgrade to 8 for newer features (parity, not required) |
| `hidden: true` frontmatter for stubs | ✅ | ❌ | Consider for v0.3.31+ — enables stub-vs-content split |
| Two-dir pattern (skills/ + skill-data/) | ✅ | ❌ | Consider for v0.3.31+ — allows external scanners to see content without binary embedding |
| `skill get --all` flag | ✅ | ❌ | v0.3.31+ candidate (matches `npx skills add` workflow) |
| `skill get --full` flag | ✅ | ❌ | v0.3.31+ candidate (loads `references/` + `templates/`) |
| Env var override (`AGENT_BROWSER_SKILLS_DIR`) | ✅ | ❌ | v0.3.31+ candidate (debugging aid) |
| `references/` and `templates/` per-skill subdirs | ✅ | ❌ | v0.3.31+ candidate (richer skill content) |
| YAML frontmatter parser (custom, not serde_yaml) | ✅ | N/A | We don't need it — skill content is opaque to the CLI |
| JSON output mode for `list`/`get`/`path` | ✅ | ❌ | Already have `--json` on most commands; add to `skill list` for v0.3.31+ |

**v0.3.31+ feature list, prioritized:**
1. **Stub-vs-content split** (H1 resolution) — most impactful single change
2. **`skill list --json`** — easy add, useful for agent tooling
3. **`skill get --all`** — useful for "load everything" workflows
4. **YAML frontmatter parser + `hidden: true`** — required for #1
5. **`references/` and `templates/` support** — requires #1
6. **Env var override** — debugging aid
7. **Upgrade rust-embed 6 → 8** — parity with agent-browser

That's roughly 4-6 hours of work spread over 2-3 PRs. Not for v0.3.30.
