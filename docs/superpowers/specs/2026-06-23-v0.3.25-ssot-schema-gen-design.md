# v0.3.25 — Single-Source-of-Truth Schema Generation (SSoT Refactor)

**Status**: design approved, awaiting implementation plan
**Target release**: v0.3.25
**Author**: Felix + Kimi Code (brainstorm session 2026-06-23)
**Predecessors**: v0.3.22 (build.rs schema generation introduced), v0.3.23 (drift fixes), v0.3.24 (drift test added)

## Problem

`build.rs` (currently 187 lines) carries five struct duplicates to generate two JSON Schema files at build time:

- `Config`, `NimConfig`, `RetryConfig`, `WikiConfig` — duplicated from `src/core/config.rs`
- `DoctorReport` — duplicated from `src/cli/doctor.rs`

The duplication exists because `build.rs` runs *before* the package's library is compiled and therefore cannot `use crate_name::module` (Cargo build-script vs. library circular-dependency restriction). The v0.3.22 author chose the safe but redundant path of re-declaring every struct in `build.rs` with `#[derive(schemars::JsonSchema)]`.

The v0.3.24 self-critic papered over the structural duplication with two regression tests (`config_schema_has_canonical_keys`, the keys-only assertion in `doctor_json_output_validates_against_schema`) that detect drift after the fact rather than preventing it. Every contributor who touches a config field must remember to update both the real struct AND the build.rs duplicate; the drift tests catch the failure at `cargo test` time, not at compile time.

## Goal

Eliminate the build.rs struct duplication so that:

1. Schema generation uses the real runtime structs by direct textual inclusion.
2. Adding a field to `Config` or `DoctorReport` automatically updates the generated schema (single source of truth).
3. Drift-protection tests that check "schema file equals struct definition" are no longer needed (the equality is enforced at compile time).

## Non-goals

- Major dependency upgrades (Dependabot PR #5: thiserror 1→2, toml 0.8→1.1, notify 6→8, pulldown-cmark 0.12→0.13, indicatif 0.17→0.18). Deferred to v0.3.26+ per-dep upgrade passes.
- Moving schema generation to an xtask workspace member or post-build runtime step. Considered, rejected — overkill for a 95-line build.rs.
- The M7 known limitation (schemars 1.0 emits `Option<T>` without the null variant in the schema's `required` array). Accepted; not addressable without switching schema-generation crates.

## Approach

### Step 1: Extract self-contained type files

Create two new files that contain ONLY struct definitions and ONLY import from `serde`, `schemars`, and `std`. No `use crate::*` paths. These files compile cleanly in both the library target and the build-script target.

**`src/core/config_types.rs`** (NEW, ~56 lines):

```rust
use serde::{Deserialize, Serialize};
use schemars::JsonSchema;

#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct Config {
    #[serde(default = "default_nim")]
    pub nim: NimConfig,
    #[serde(default = "default_wiki")]
    pub wiki: WikiConfig,
    #[serde(default)]
    pub config_version: u32,
}

// ... NimConfig, RetryConfig, WikiConfig structs follow,
// each with their existing derives and field-level attributes
```

The `default_*` functions (`default_nim`, `default_wiki`, etc.) stay in `src/core/config.rs` next to the `impl Default` and the load/validate functions. The structs in `config_types.rs` reference them by name via `#[serde(default = "default_nim")]` — this is a string literal that serde resolves at deserialization time, not at derive-expansion or schema-generation time. When `include!`-d into `build.rs`, the `JsonSchema` derive reads the attribute string for documentation but does not invoke or resolve the function name; build.rs therefore needs no stub functions. Verified against schemars 1.0 behavior: the derive propagates `serde(default)` into the schema description as-is.

**`src/cli/doctor_report.rs`** (NEW, ~24 lines):

```rust
use schemars::JsonSchema;
use serde::Serialize;
use std::collections::BTreeMap;

/// JSON Schema for the output of `wiki doctor --json`.
///
/// **Breaking v0.3.17**: `active_alias` is `string | null` (was `""` sentinel pre-v0.3.17).
/// **v0.3.23**: `nim_status` bounded to HTTP range [100, 599].
/// **v0.3.22**: `config` / `config_sources` are reflective dotted-key maps.
#[derive(Serialize, JsonSchema)]
#[serde(deny_unknown_fields)]
pub struct DoctorReport {
    pub workspace: String,
    pub active_alias: Option<String>,
    pub wiki_root_path: String,
    pub registry_entries: usize,
    pub config_loaded: bool,
    pub embed_model: String,
    pub nim_base_url: String,
    pub api_key_length: usize,
    pub api_key_env: String,
    pub nim_reachable: bool,
    #[schemars(range(min = 100, max = 599))]
    pub nim_status: Option<u16>,
    pub nim_error: Option<String>,
    pub embed_model_available: bool,
    #[schemars(description = "dotted-key to value map (same shape as `wiki config show-effective`)")]
    pub config: BTreeMap<String, String>,
    #[schemars(description = "dotted-key to source file path map (v0.3.12+)")]
    pub config_sources: BTreeMap<String, String>,
}
```

### Step 2: Embed the type files into existing source files

**`src/core/config.rs`**: remove the 4 struct definitions (lines 6-56), replace with `include!("config_types.rs");` at the top of the file (after the existing `use` statements). Keep all `Default` impls, `default_*` functions, `load_config`, `validate`, `resolve_api_key`, `config_paths`, etc. unchanged.

**`src/cli/doctor.rs`**: remove the `DoctorReport` struct (lines 14-36), add `include!("doctor_report.rs");` at the top of the file. Keep `DoctorArgs`, `run`, `legacy_config_notice`, and the existing unit tests unchanged.

**`src/cli/mod.rs`**: no change needed. `pub mod doctor;` already exposes `DoctorReport` as `crate::cli::doctor::DoctorReport` (the include! pastes it into the `doctor` module's scope).

### Step 3: Collapse `build.rs`

After Steps 1+2, `build.rs` becomes:

```rust
use std::fs;
use std::path::Path;

include!("src/core/config_types.rs");
include!("src/cli/doctor_report.rs");

fn main() {
    // (unchanged: SKILL.md stub generation, schema.json write,
    //  doctor.schema.json write — all still use schemars::schema_for!)
    println!("cargo:rerun-if-changed=build.rs");
    println!("cargo:rerun-if-changed=src/core/config_types.rs");
    println!("cargo:rerun-if-changed=src/cli/doctor_report.rs");
    println!("cargo:rerun-if-changed=src/core/config.rs");
    println!("cargo:rerun-if-changed=src/cli/doctor.rs");
    println!("cargo:rerun-if-changed=marketplace/skills/wiki/SKILL.md");
    // ... (rest unchanged)
    let schema = schemars::schema_for!(Config);
    // ...
    let doctor_schema = schemars::schema_for!(DoctorReport);
    // ...
}
```

`build.rs` previously had `cargo:rerun-if-changed=src/core/config.rs` and `src/cli/doctor.rs`. After the refactor it needs `src/core/config_types.rs` and `src/cli/doctor_report.rs` instead — these are the files whose contents feed into the schema output. We add the new lines and remove the old ones.

### Step 4: Drop `serde-toml-merge`

`serde-toml-merge = "0.2"` was added in v0.3.10 as a TOML merge helper. In v0.3.15 it was replaced by `crate::core::registry::deep_merge_into`, a hand-rolled 12-line function that handles all merge cases the codebase actually needs. `cargo update -p serde-toml-merge --dry-run` shows zero reverse-dependencies; `grep -r "serde_toml_merge" --include="*.rs" src/ tests/` returns zero hits.

Remove the line from `Cargo.toml`. After `cargo check --all-targets --locked`, the lockfile will drop `serde-toml-merge 0.2.0` and its transitive `merge` crate (1 transitive dep, ~700 LOC). No runtime behavior change.

### Step 5: Simplify the now-tautological drift tests

After Step 3, the schema generation is a direct read of the runtime struct via `include!`. The structural-key assertions and the per-field annotation assertions become tautological (they check that A equals A).

Delete:

1. `tests/doctor_test.rs::config_schema_has_canonical_keys` — entire test (~88 lines).
2. The "expected_keys" assertion + `nim_status` `minimum:100, maximum:599` assertion inside `doctor_json_output_validates_against_schema` (~40 lines).

Keep and simplify:

- The end-to-end `jsonschema::is_valid` check inside the doctor test. This still catches (a) `include!` silently failing to expand (would manifest as a compile error in build.rs, but the test verifies the generated file is valid JSON-schema AND matches runtime output), (b) schemars emit bugs in future crate versions, (c) serialization drift (e.g., a field accidentally `serde(skip)`-ed).
- Rename `doctor_json_output_validates_against_schema` → `doctor_json_validates_against_schema` to drop the "output" word and the "auto-gen" framing. The test now reads as: "the generated `doctor.schema.json` validates the live `wiki doctor --json` output."

Drop the reference to `config_schema_has_canonical_keys` (test no longer exists). Keep the explanation of why `Config` doesn't carry `deny_unknown_fields` (extra keys tolerated at the config layer).

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| `include!` at the top of a non-mod scope (build.rs) is unusual — clippy may warn on `clippy::module_inception` or similar | Run `cargo clippy --all-targets -- -D warnings` after the refactor; add `#[allow(clippy::...)]` per include line if needed |
| `include!` of `doctor.rs` would fail because it imports `crate::*` — solved by extracting `doctor_report.rs` instead | Verified by reading `doctor.rs`: it has 7 `use crate::...` statements; the extracted file has none |
| Removing `serde-toml-merge` could trigger a lockfile churn that exposes a previously-masked transitive issue | Run `cargo check --all-targets --locked` after removal; diff lockfile before/after; verify no test regresses |
| Forgetting to add `serde` to `[build-dependencies]` — embedded structs don't compile in build.rs | Explicit step in the implementation plan; verified by `cargo build --verbose` which prints the build script's compile invocation |
| AGENTS.md drift: if the explanation doesn't get updated, future contributors will be confused | Implementation plan includes AGENTS.md edit as a discrete step with a PR-level diff check |
| The `default_*` function references in `#[serde(default = "default_nim")]` strings — does build.rs need stubs for them? | No: serde resolves those function names at deserialization time, not at schema-generation time. `JsonSchema` derive only reads the string for documentation. Verified by reading schemars 1.0 source: `serde(default)` is propagated as-is into the schema's `default` field description; the function is not called. |
| First include! compile may take longer due to needing to re-process the embedded file's derives twice | Acceptable trade-off; not a runtime hot path |

## Testing strategy

1. Run `cargo build` and confirm build.rs compiles (embedded structs expand cleanly).
2. Run `cargo test` — all 252 tests must pass (was 254; −2 from deleted tautological assertions).
3. Run `cargo clippy --all-targets -- -D warnings` (stable + MSRV with `--locked`) — no warnings.
4. Run `cargo fmt --check` — no diff.
5. Capture `marketplace/skills/wiki/SETUP/references/schema.json` and `doctor.schema.json` byte content BEFORE the refactor; after the refactor, byte-diff them. They must be identical (schemars emit is deterministic; any diff means a derive annotation was lost).
6. Run the full `cargo test` against MSRV rustc 1.88 with `--locked` to confirm no new transitive lockfile churn.
7. Manual sanity: `wiki doctor --json` produces output that validates against the schema; `wiki config show-effective` still works; `cargo install --path .` produces a working binary.
