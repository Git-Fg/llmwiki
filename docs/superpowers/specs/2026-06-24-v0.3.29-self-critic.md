## Release notes (for GitHub)

### v0.3.29 — Flat skill bundle

**Removed — `marketplace/` directory and all plugin-manifest plumbing:**
- Deleted `marketplace/` (Claude/Kimi/Cursor/Codex plugin manifests, `marketplace.json`, `VERSION`, `validate.py`).
- Deleted legacy `agents/skills/` build-artifact stub.
- Removed `marketplace-validate` CI job and `python3 marketplace/scripts/validate.py --strict` PR template step.

**Why:** the marketplace layer added packaging complexity without improving agent outcomes. Flat `skills/wiki-{topic}.md` + one `~/.agents/skills/wiki/SKILL.md` hub is enough. Inspired by `vercel-labs/agent-browser`'s `skills get core` convention: CLI is source of truth, skill is a thin discovery hub.

**Added — `rust-embed` skill bundle:**
- `src/skills/mod.rs` replaces 9 hand-written `include_str!` constants with one `#[derive(RustEmbed)] #[folder = "skills/"]` over the source-of-truth `skills/` directory.
- `find_skill("search")` and `find_skill("wiki-search")` both work (name normalization).
- `list_skills()` returns the 9 inline sub-skills, excluding the hub.
- New dep: `rust-embed = "6"`.

**Added — `wiki skill get <topic>` canonical discovery primitive:**
- Mirrors `agent-browser`'s `skills get` convention.
- Accepts both `wiki-search` and `search`.
- `wiki skill show` retained as alias.
- New `wiki skill path [topic]` prints install location.

**Added — `llmwiki-cli install-skill --global` is hub-only:**
- Writes **only** `~/.agents/skills/wiki/SKILL.md`.
- Sub-skills are served at runtime from bytes embedded in the binary — never written to disk.
- Reduces install surface from 9 files + 1 hub to **1 file**.

**Rewritten — `skills/SKILL.md` (hub, 109 lines):**
- Frontmatter + gotchas + how-to, routing every command to `llmwiki-cli <cmd> --help` for the full reference.
- Explicit "always prefer `llmwiki-cli skill get` over guessing commands" rule.
- All CLI references use `llmwiki-cli`, never `wiki`.
- Lists the 9 inline sub-skills and how to fetch each.

**Rewritten — 9 inline sub-skills (40-48 lines each, was 35-97):**
- `wiki-setup`, `wiki-config`, `wiki-ingest`, `wiki-search`, `wiki-query`, `wiki-lint`, `wiki-models`, `wiki-sync`, `wiki-troubleshooting`.
- Each has `name`, `description`, `allowed-tools` frontmatter (Claude/Kimi compatible).
- Each is thin: routing header + "Use when / Do NOT use for" guidance + 3-5 key commands.
- Full flag reference is in `llmwiki-cli <cmd> --help`.

**Migration:**
```bash
llmwiki-cli install-skill --global   # overwrites hub with v0.3.29
# Optional — clean up v0.3.28 nested folders:
rm -rf ~/.agents/skills/wiki/{SETUP,INGEST,SEARCH,QUERY,LINT,MODELS,SYNC,TROUBLESHOOTING,CONFIG,LSP,MCP}
```

## Self-critic — what the implementation does well

1. **Hub is genuinely minimal** — 109 lines, no embedded workflows, pure discovery router. Matches agent-browser precedent (149 lines, defers to `skills get core`).
2. **Sub-skills don't duplicate `--help`** — only "Use when / Do NOT use for" + 3-5 key commands. Flag tables removed. Matches spec §4.6 trim pattern.
3. **Version-matched content** — sub-skills served from binary bytes, never go stale. Matches agent-browser's "always matches the installed version" guarantee.
4. **Single source of truth** — `skills/` is the only canonical location; rust-embed + build.rs (schema.json) generate everything else.
5. **Hub-only install** — `remove_dir_all` cleans up v0.3.28 nested folders automatically on next install.
6. **Name normalization** — both `wiki-search` and `search` work, no surprise to users.
7. **Verification gates are all green** with fresh evidence — not extrapolated from prior runs.

## Self-critic — what needs attention

### 🔴 HIGH — Spec-vs-implementation gap on discovery

**Spec §2.1 Goal #1 says:**
> "Each sub-skill is a **top-level** entry in `~/.agents/skills/wiki-{name}/SKILL.md`. An agent that lists `~/.agents/skills/` sees 10 entries: the hub `wiki` and the 9 siblings."

**Spec §4.1 layout shows:**
```
~/.agents/skills/
├── wiki/SKILL.md                    (≤80 lines, thin stub)
├── wiki-setup/SKILL.md              (top-level)
├── wiki-config/SKILL.md
... (9 sibling directories)
```

**What the implementation does:** only `wiki/SKILL.md` is installed. The 9 sub-skills are served at runtime via `wiki skill get`. An agent that lists `~/.agents/skills/` sees **1 entry, not 10**.

**Why this happened:** the user message during execution was:
> "the marketplace should be only containing the small really short skill telling how/when to use the cli and auto discover capacitis + lit of skills get available capacities and when to use each"

This is **internally in tension** — "small really short skill" implies 1 file, but "auto discover capacitis" implies auto-discovery (10 entries). I picked the "small install" interpretation. The spec picked the "auto-discovery" interpretation.

**Research evidence:** Anthropic's [Agent Skills spec](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview) and Kimi Code's [skills docs](https://moonshotai.github.io/kimi-cli/en/customization/skills.html) both describe a **recursive filesystem scan** model:
> "Recursively scan ~/.claude/skills/ (and .claude/skills/) for directories containing SKILL.md files, regardless of nesting depth."

Kimi supports an additional **flat `.md` form** — `~/my-skills-collection/demo-ui-components.md` → name `demo-ui-components`. This would let the agent discover all 9 sub-skills from filesystem scan without 9 nested directories.

**Recommendation:** user-decision. Three options:
- **A. Keep current design** (1 hub on disk, 9 served on demand) — matches "small install" interpretation; matches the user's "small really short skill" framing; cost: agent must invoke `wiki skill list` once before seeing the 9 sub-skills.
- **B. Install 10 top-level dirs** (`wiki-setup/`, `wiki-config/`, ...) — matches spec Goal #1; matches auto-discovery; cost: 10 dirs vs 1.
- **C. Install 9 flat `.md` files** (`~/.agents/skills/wiki-search.md`, ...) — matches Kimi's flat .md form; matches auto-discovery; cost: 9 files vs 1, but still host-agnostic.

I'd recommend **C** if the user wants auto-discovery AND minimal install surface. It's also the most host-agnostic.

### 🟡 MEDIUM — Hub is 109 lines, spec target is ≤80

**Spec §2.1 Goal #2:** ≤80 lines.
**Actual:** 109 lines.

The excess is two tables:
- "Internal command summary" (lines 62-77, 16 lines) — duplicates `--help` tersely.
- "Inline sub-skill index" (lines 97-109, 13 lines) — complements the description.

Both are user-endorsed "steering into correct capacities" content, but the count is over budget. Could trim by removing the command summary table (rely on sub-skill index + auto-discovery), saving 16 lines → 93 lines. Still over 80.

### 🟡 MEDIUM — `cargo:rerun-if-changed=skills/SKILL.md` is redundant

Build.rs manually lists `skills/SKILL.md` as a rerun trigger. rust-embed's proc macro already emits `cargo:rerun-if-changed` for every file in the embedded folder. So our manual entry is redundant (harmless, but signals incomplete understanding). The build still works correctly because rust-embed handles it.

### 🟢 LOW — Discovery of sub-skills is one CLI hop away

This is the core trade-off of Option A above. Agents that follow `wiki skill list` from the hub will discover all 9 sub-skills in <1 second. But it's not zero-hop. Worth documenting the trade-off in CHANGELOG under "How sub-skills work" section.

Archived 3 files (1 plan, 2 specs) that referenced the removed marketplace / LSP / MCP — they're no longer relevant to forward development. Available in `docs/superpowers/{plans,specs}/_archive/` for historical context.

## What the spec didn't predict

- **rust-embed** — the spec assumed `include_str!` (already in use pre-v0.3.29). Switch to rust-embed happened mid-execution when it became clear that `include_str!` doesn't compose well with `find_skill()` runtime discovery. Net win: 80% less boilerplate.
- **`wiki skill path` subcommand** — added mid-execution; not in spec. Useful for debugging "where does this skill actually come from?".
- **`remove_dir_all` cleanup** — install-skill now nukes the whole `wiki/` dir before recreating. Spec §8 risk table said "install-skill does *not* delete orphan dirs; document in CHANGELOG"; we went the other way (auto-cleanup). Better UX.
