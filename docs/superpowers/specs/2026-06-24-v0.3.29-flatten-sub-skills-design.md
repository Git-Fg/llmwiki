# v0.3.29 — Flatten sub-skills, slim hub, stop duplicating `--help`

**Status:** Draft, in progress.
**Date:** 2026-06-24
**Author:** Brainstorm session informed by `vercel-labs/agent-browser` methodology (the gold standard for AI-agent CLIs in 2026).
**Target llmwiki version:** v0.3.29 (minor, follows v0.3.28 LSP/MCP removal).
**Predecessor:** [2026-06-24-v0.3.28-remove-lsp-mcp-design.md](2026-06-24-v0.3.28-remove-lsp-mcp-design.md).

---

## 1. Problem statement

After v0.3.28 dropped the embedded LSP/MCP servers, the remaining skill bundle is the *only* agent-facing surface. A direct review of that bundle against the `vercel-labs/agent-browser` methodology (the consensus gold standard for AI-agent CLIs in 2026) surfaces three concrete defects:

1. **Sub-skills are hidden from auto-discovery.** An agent that scans `~/.agents/skills/` enumerates `wiki/` (the hub) and stops. The 9 sub-skills (`SETUP/`, `CONFIG/`, `INGEST/`, `SEARCH/`, `QUERY/`, `LINT/`, `MODELS/`, `SYNC/`, `TROUBLESHOOTING/`) live physically inside the hub bundle. An agent that never loads the wiki hub will never see them. `llmwiki-cli skill list` enumerates them, but the agent has to know to run that command first — the hub only mentions it on line 64, after five other sections of context.

2. **Sub-skills duplicate `wiki <cmd> --help`.** `SEARCH/SKILL.md` and `QUERY/SKILL.md` each include a `## Flags` table with the same flag names and defaults that `wiki search --help` and `wiki query --help` already emit. When the CLI changes a default (e.g. `--top-k` default changes from 5 to 8), the skill stays stale. The skill should *steer* the agent into the right command, not *pre-document* the command — that's what `--help` is for.

3. **Hub is bloated.** The hub SKILL.md is 94 lines and embeds an edit-config workflow (lines 46-60) that belongs entirely in the CONFIG sub-skill, plus a "Quick commands" table (lines 77-87) that duplicates every sub-skill's workflow, plus a "Multiple wikis" section (lines 89-94) that overlaps with SYNC. The hub should be a thin discovery stub — like `agent-browser`'s 149-line hub that defers to `skills get core`.

The user's framing in chat: *"why are the SETUP / SEARCH / QUERY skill within the main one? Those should be auto discoverable skills through use of like llmwiki skill get commands. … don't duplicate internal cli skills or at least not too much (steering into using correct capacities is great though)."*

## 2. Goals & non-goals

### 2.1 Goals

1. Each sub-skill is a **top-level** entry in `~/.agents/skills/wiki-{name}/SKILL.md`. An agent that lists `~/.agents/skills/` sees 10 entries: the hub `wiki` and the 9 siblings.
2. The hub SKILL.md is **≤ 80 lines**, purely a discovery router. No embedded workflows.
3. Sub-skill SKILL.md files **never duplicate `wiki <cmd> --help`** content. They steer the agent into the right command, then point at `--help` for the full flag reference.
4. `llmwiki-cli skill get <name>` is the canonical discovery primitive (alias of existing `skill show`).
5. `wiki --help` and `wiki <cmd> --help` are **self-guiding for AI agents** (categorized commands, one-line usage examples, "Start here for AI agents" pointer). Implemented via clap `long_about` / `after_help` doc comments — no new flag plumbing required.
6. `wiki install-skill --global` creates the 10 top-level directories atomically.

### 2.2 Non-goals (deferred to v0.3.30+)

- MCP server for `llmwiki-cli` (out of scope per v0.3.28 decision — skills + CLI is sufficient).
- Renaming `llmwiki-cli` → something shorter. The binary name is a noun; it's fine.
- Per-workspace skill install layout change. `--workspace` continues to install to `<workspace>/.agents/skills/` with the same flat top-level layout.
- Auto-install via `curl | sh` from the skill (Phase 3 of v0.3.0 plan, deferred indefinitely).
- Locale-specific sub-skill translations.

## 3. Reference methodology — `vercel-labs/agent-browser`

Studied `~/.agents/skills/agent-browser/SKILL.md` (149 lines) and `agent-browser --help` (categorized commands, one-line examples) on this machine. Three principles:

1. **Hub is a stub.** ~150 lines. Just enough to tell you *what* the tool is and *what command to run first*. All workflow content lives in skills that the CLI serves dynamically.
2. **CLI is the source of truth.** `agent-browser skills get core` returns version-matched content. The README says: *"The stub is intentionally minimal — it points Claude Code at `agent-browser skills get core`. … The CLI serves skill content that always matches the installed version, so instructions never go stale."*
3. **Auto-discovery via `skills list`.** Specialized skills are siblings. Any agent that runs `agent-browser skills list` discovers them all.

We adopt (1) and (3) directly. We adopt a softer version of (2): the binary still bundles the sub-skill content via `include_str!` (already the case), so installation is offline-first; `skill list` + `skill show` are the discovery primitives.

**After (v0.3.29):**

├── wiki-setup/SKILL.md                   (top-level)
├── wiki-config/SKILL.md

Same change in `marketplace/skills/` (source-of-truth) and the build-regenerated `agents/skills/wiki*/` stubs.

### 4.2 Sub-skill frontmatter rename

Each sub-skill's `name:` field changes from `<topic>` to `wiki-<topic>`:

```yaml
# Before
name: setup

# After
name: wiki-setup
```

The marketplace validator already enforces `meta["name"] == path.parent.name.lower()`. After the move, `path.parent.name` is `wiki-setup`, so the `name:` field must match. Validator update is mechanical.

### 4.3 `skill list` output — line counts already correct

Current `skill list` output:

search               (36 lines)

lint                 (49 lines)

After rename:

```

Sub-skill line counts will *drop* (target: 25-40 lines each, after trimming duplicated flag tables). No source code change required — `src/skills/mod.rs` already uses `include_str!` keyed by topic name; the rename is content-only.

### 4.4 New `skill get` subcommand

Add `llmwiki-cli skill get <topic>` as an alias of `skill show <topic>`. Reason: `agent-browser` uses `skills get`, and the user's chat explicitly suggested "llmwiki skill get commands". We keep `show` for backward compatibility; `get` is the canonical discovery primitive going forward.

```rust
// src/cli/skill.rs — add to Command enum
enum SkillCmd {
    Show { topic: Option<String> },  // existing
    Get { topic: Option<String> },   // NEW — alias of Show
    List,                            // existing
    Path { topic: Option<String> },  // existing
}
```

Implementation: dispatch `Get` to the same handler as `Show`. One trivial code change.

### 4.5 Self-guiding `--help`

Clap already supports `long_about` and `after_help` doc comments on each `#[command(...)]` and `#[arg(...)]`. We add:

- **Top-level `llmwiki-cli --help`**: `after_help` with categorized command list and "Start here for AI agents: `llmwiki-cli skill list`" pointer.
- **Per-command `--help`** (where useful): `long_about` with a 2-3 sentence workflow tip and an example.

This is doc-comment work — no behavior change, no flag change. Tests don't need to update.

### 4.6 Sub-skill content trim

Each sub-skill SKILL.md is rewritten to remove any content that duplicates `wiki <cmd> --help`. The trim pattern:

```markdown
# Before (SEARCH/SKILL.md)
## Flags
- `--top-k N` — number of results (default 5)
- `--threshold 0.0-1.0` — minimum similarity (default 0.0)
- `--model <name>` — override the embedding model
- `--json` — machine-readable output
- `--wiki <alias>` — search a different wiki without `cd`

# After
## Flags

Run `llmwiki-cli search --help` for the full flag reference.

## Tips
- Lower `--threshold` = more results; raise to 0.5+ for high-precision
- Combine with `llmwiki-cli tree | grep` for slug-based filtering before search
- Embeddings must exist first: `llmwiki-cli embed`
```

Workflow tips stay (they're not in `--help`). Flag tables go (they're 100% duplicated).

### 4.7 Hub slim-down

Target: ≤ 80 lines. Structure:

```markdown
---
name: wiki
description: |
  Personal Karpathy-style LLM Wiki. Use when the user asks to ingest
  a source, search the wiki, answer a question against prior research,
  lint or maintain the wiki, set up on a new device, or pick a
  different NVIDIA NIM embedding model. Prefer the wiki's native
  file tools for browsing; reach for `llmwiki-cli` only when
  semantic search or NIM-backed operations are needed.
---

# wiki

Personal Karpathy-style LLM Wiki — markdown + JSONL embeddings, no database.

## Start here (for AI agents)

```bash
llmwiki-cli skill list                 # see all sub-skills
llmwiki-cli skill get <name>           # load one sub-skill
llmwiki-cli <command> --help           # full flag reference
```

## When to use `llmwiki-cli` vs native file tools

Use the CLI for: semantic search, RAG queries, ingest, NIM diagnostics.
Use native file tools for: browsing pages, reading the catalog, diffing.

## Filesystem layout

[12 lines: wiki-root.toml, .agents/skills/wiki/, <workspace>/{wiki/,raw/,index.md,log.md,embeddings.jsonl}]

## Sub-skills

| Topic | Use when |
|-------|----------|
| `wiki-setup` | First-run install, init, registry, skill install |
| `wiki-config` | Edit wiki-root.toml or per-workspace config.toml |
| `wiki-ingest` | Adding a source file |
| `wiki-search` | Semantic search by vector similarity |
| `wiki-query` | RAG question-answering with citations |
| `wiki-lint` | Hygiene checks, broken links, frontmatter |
| `wiki-models` | Switch embedding/reranking model |
| `wiki-sync` | New-device setup, tailnet git sync |
| `wiki-troubleshooting` | Diagnose wiki errors |
```

Removed sections (moved to sub-skills or deleted):
- "Before editing any config" → moved entirely into `wiki-config`
- "Quick commands" table → deleted (duplicates every sub-skill)
- "Multiple wikis" → moved into `wiki-sync`
