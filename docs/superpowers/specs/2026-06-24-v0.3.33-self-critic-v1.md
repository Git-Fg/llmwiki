## The Architecture Pivot — Why It Was Necessary

Pre-v0.3.33, the same sub-skill content lived in TWO places:

```
skills/SETUP/SKILL.md     ← copied to ~/.agents/skills/wiki/SETUP/SKILL.md
src/skills/data/SETUP.md  ← embedded in binary via rust-embed, served by `skill get`
```

These could drift apart silently. v0.3.33 collapses to ONE source: `src/skills/data/`. The hub at `skills/SKILL.md` is the single public-facing file; everything else is CLI-internal.

**Compile-time guarantee:** if a sub-skill file is added or removed from `src/skills/data/`, the binary rebuilds automatically (rust-embed emits `cargo:rerun-if-changed=` lines for each file).

**Install-time guarantee:** `npx skills add Git-Fg/llmwiki` installs **1 file** (was 11). Less attack surface, less drift risk, faster install.

## End-of-Iteration Reflection

v0.3.33 is the simplest published version of llmwiki yet. The hub is 40 lines, the binary embeds 9 sub-skills, `npx skills add` installs 1 file. The drift risk that motivated v0.3.29 ("sub-skills duplicate `wiki <cmd> --help`") is now eliminated at a deeper level: sub-skills can't drift because they only exist in one place.

**What worked:** The "single source of truth" principle applied rigorously. The hub is now so simple that there's almost nothing to break.

**What I'd do differently:** None for v0.3.33. The spec violation (`name: wiki` in `skills/`) is a known-deferred item and stays deferred.
