## H1 update — discovery layout decision

**Research findings (R1):**
- [agentskills.io spec](https://agentskills.io/specification): canonical form is `<name>/SKILL.md` (directory); `name:` frontmatter **must match parent dir name** (strict).
- [Kimi Code docs](https://moonshotai.github.io/kimi-cli/en/customization/skills.html) explicitly support an additional **flat `.md` form** — `wiki-search.md` → name `wiki-search`. Not in the canonical spec.
- [Claude Code docs](https://code.claude.com/docs/en/skills): directory form only, no flat `.md` support mentioned.

**Research findings (R2):**
- `vercel-labs/agent-browser`'s own pattern (the v0.3.29 reference methodology) is **single `SKILL.md` in `<name>/`** — i.e., the directory form. Matches my v0.3.29 install (1 hub at `~/.agents/skills/wiki/SKILL.md`).
- Sub-skills are served on-demand by the binary via `agent-browser skills get core` — not pre-installed on disk.

**Research findings (R3):**
- Claude API Skills are **sandboxed** (uploaded, not filesystem-based). Out of scope for our `install-skill`.
- Claude Code reads from `~/.claude/skills/`, NOT `~/.agents/skills/`.
- [Cross-host convention map](https://medium.com/google-cloud/confused-about-where-to-put-your-agent-skills-ea778f3c64f3):
  - Kimi: `~/.agents/skills/` (generic fallback) + brand paths
  - Claude Code: `~/.claude/skills/`
  - Cursor: `~/.cursor/skills/`
  - Gemini CLI: `~/.gemini/skills/`
  - Antigravity global: `~/.gemini/config/skills/`
  - Codex: `~/.codex/skills/`

**Implication for the H1 decision (A/B/C):**

| Layout | Compatible with | Files installed |
|---|---|---|
| A. Current (1 hub at `~/.agents/skills/wiki/`) | Kimi only | 1 file |
| B. 10 top-level dirs (`~/.agents/skills/wiki-{topic}/SKILL.md`) | Kimi only | 10 files + 10 dirs |
| C. 9 flat `.md` files (`~/.agents/skills/wiki-{topic}.md`) | Kimi only (flat .md is Kimi-specific) | 10 files |
| D. 1 hub at `~/.claude/skills/wiki/` (spec-strict brand) | Claude Code + Kimi (Kimi reads brand path) | 1 file |
| E. Multi-brand: install to `~/.claude/skills/` AND `~/.agents/skills/` AND `~/.kimi/skills/` | Claude Code + Kimi | 3 files + symlink dance |
| F. New `--install-path <dir>` flag (user picks) | Whatever the user wants | 1-N files |

**Recommendation:** ship H1 as **A** (current design, works for the user's apparent host = Kimi) and add **F** (`--install-path` flag) for v0.3.30 so users on other hosts can install to the path their agent actually reads. B and C are over-engineering (10 files when 1 file + the runtime `skill get` primitive does the job). D and E are good for the multi-host case but need a host-detection mechanism to avoid installing everywhere by default.

**Why A over B/C:** the user explicitly endorsed "small really short skill" + "auto discover capacitis + lit of skills get available capacities". A fulfills both: small install (1 file), auto-discovery via `wiki skill list` (one CLI hop). B/C add 10 files for marginal benefit (auto-discovery without the CLI hop — but the cost is the install surface).

## H2 new finding — cross-host fragmentation

**The problem:** `llmwiki-cli install-skill --global` writes to `~/.agents/skills/wiki/SKILL.md`. This path is:
- ✅ Read by Kimi Code (the user's apparent host)
- ❌ NOT read by Claude Code (which reads `~/.claude/skills/`)
- ❌ NOT read by Cursor (which reads `~/.cursor/skills/`)
- ❌ NOT read by Gemini CLI (which reads `~/.gemini/skills/`)

**For Kimi users:** works as designed. Skill is visible immediately.

**For Claude Code / Cursor / Gemini users:** the install is silent — the file is written but never loaded by their agent. They'd need to:
1. Configure their agent to read from `~/.agents/skills/`, OR
2. Symlink: `ln -s ~/.agents/skills ~/.claude/skills`, OR
3. Manually re-install to the right path.

**Evidence:** confirmed by reading [Claude Code Skills docs](https://code.claude.com/docs/en/skills), [Kimi Code skills docs](https://moonshotai.github.io/kimi-cli/en/customization/skills.html), and the [Dazbo cross-host guide](https://medium.com/google-cloud/confused-about-where-to-put-your-agent-skills-ea778f3c64f3).

**Recommendation:** add `--install-path <dir>` flag to `install-skill` for v0.3.30. Users on different hosts can point to their agent's skills dir. Default stays `~/.agents/skills/wiki/` (the cross-host "generic" convention + the user's existing pattern).

```bash
llmwiki-cli install-skill --global                       # default: ~/.agents/skills/wiki/
llmwiki-cli install-skill --global --install-path ~/.claude/skills   # Claude Code
llmwiki-cli install-skill --global --install-path ~/.cursor/skills   # Cursor
llmwiki-cli install-skill --global --install-path ~/.gemini/skills   # Gemini CLI
```

Effort: ~30 minutes (add arg, pass through, document).
