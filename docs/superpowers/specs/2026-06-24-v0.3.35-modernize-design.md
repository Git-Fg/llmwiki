# v0.3.35 — Modernization: typed Frontmatter + selective lint + cargo-deny

**Status:** Design — awaiting user approval before plan + implementation.
**Date:** 2026-06-24
**Target release:** v0.3.35
**Branch:** `release/v0.3.35-modernize` (finalized in plan)

## Motivation

The v0.3.34 release (`eec848c`) shipped dep cleanup + low-risk bumps (thiserror 1→2, rust-embed 6→8, jsonschema 0.18→0.46) and removed 3 dead direct deps. Two modernization items remain:

1. **`serde_yaml` 0.9 is archived.** GitHub: <https://github.com/dtolnay/serde-yaml> "This project is no longer maintained." RUSTSEC-2025-0068 advisory still applies to its C-FFI `libyaml` backend. The user-facing impact is small (we only call `serde_yaml::Value` and `serde_yaml::from_str` in `src/core/markdown.rs`), but every release inherits the dependency-of-an-archived-crate posture.
2. **The frontmatter is dynamic `serde_yaml::Value`.** Every wiki page parses through `parse_frontmatter` and downstream consumers (cli::ls, cli::lint, cli::embed, cli::status) reach into the dynamic Value tree via `value["type"].as_str().unwrap_or("page")` and friends. Type-safety + SSOT pressure pushes toward a typed `Frontmatter` struct that is the single source of truth for: runtime parsing, the build-time-generated JSON Schema, and lint checks.

Two adjacent modernization axes the user named ("optimal lint + type safety + SSOT"):

3. **Selective clippy lints** at `warn` level (not `deny`) to catch the killer bugs (UTF-8 boundary panics, swallowed errors, panic macros) without churn on existing 289 tests.
4. **cargo-deny** for license/bans/advisories/sources policy enforcement — one config file, one CI gate.

Three candidates deferred (explicitly out of scope):

- **toml 0.8 → 1.x** — research now complete (`docs.rs/toml/latest` shows `from_str`/`Value`/`to_string` API preserved), but per-call-site audit (`src/core/registry.rs`, `src/core/models_registry.rs`) still needs to happen. Defer to v0.3.36; bundle with cargo-deny hardening + clippy nursery lints.
- **walkdir → ignore** — would change observable behavior (`.gitignore` honoring); user-visible, requires design discussion. Defer.
- **chrono → time/jiff** — current chrono 0.4.45 is fine; CVE-2020-26235 is fixed; ecosystem compat matters more than perf. Defer.

## Approach

Single v0.3.35 release bundling four coordinated changes:

1. Typed `Frontmatter` struct in `src/core/frontmatter.rs`
2. `serde_yaml` → `serde-saphyr` migration via typed-deserialize path
3. New `clippy.toml` with 4 selective lints (warn level)
4. New `deny.toml` for cargo-deny

The four axes share a single SSOT theme: typed structures (Frontmatter, deny.toml's allowlist, clippy's lint list) are the source of truth, not runtime heuristics.

### Typed Frontmatter

```rust
// src/core/frontmatter.rs
use std::collections::BTreeMap;

#[derive(Debug, Clone, serde::Deserialize, serde::Serialize, schemars::JsonSchema)]
pub struct Frontmatter {
    pub title: Option<String>,
    #[serde(default)]
    pub tags: Vec<String>,
    #[serde(rename = "type")]
    pub page_type: Option<String>,
    #[serde(default)]
    pub sources: Vec<String>,
    pub confidence: Option<f64>,
    pub created: Option<String>,
    pub updated: Option<String>,
    pub schema_version: Option<i64>,
    pub status: Option<String>,
    pub kind: Option<String>,
    pub domain: Option<String>,
    pub maturity: Option<String>,
    pub reviewed: Option<String>,
    #[serde(default)]
    pub aliases: Vec<String>,
    pub description: Option<String>,
    #[serde(default)]
    pub related: Vec<String>,
    pub source_type: Option<String>,
    pub sha256: Option<String>,
    pub ingested: Option<String>,
    pub name: Option<String>,
    #[serde(default)]
    pub descriptions: Vec<String>,
    #[serde(flatten)]
    pub extra: BTreeMap<String, serde_json::Value>,
}
```

The 21 typed fields are the keys with **clear semantic purpose that appear in at least one of the 4 real flat-layout wikis** (`mevin`, `minimax-code-wiki`, `MyWiki`, `PharmaWiki`), as confirmed by the v0.3.35 frontmatter audit (see `docs/superpowers/specs/2026-06-24-frontmatter-field-audit.md`). All 21 fields appear with ≥19 occurrences; 15 of 21 meet the original ≥50 target. The 6 fields below 50 (`related`, `source_type`, `sha256`, `ingested`, `name`, `descriptions`) are kept because each captures a distinct semantic category — cross-references, raw-ingest metadata, content hashes, slug identification, and SKILL.md-style plural descriptions — that future wikis are likely to use.

The `extra: BTreeMap<String, serde_json::Value>` flatten captures the ~30 niche fields (`avatar`, `license`, `timezone`, `requiresBeta`, `last_updated`, `session`, `schedule`, `retired`, `triggers`, etc.) used by some wikis but not others, preserving them without forcing a closed schema.

`type: Option<String>` (not `enum`) — the audit found 20+ distinct `type` values across the 4 wikis plus pipeline expressions like `"entity | concept | comparison | query | schema | summary"`. A strict enum would break 3 of the 4 wikis.

**Compatibility check for `serde_json::Value` flatten target:** serde-saphyr is serde-compatible, so `serde_json::Value` deserializes from a YAML node the same way it does from JSON. Confirmed by serde-saphyr's docs (it's a pure-Rust YAML 1.2 parser with serde integration).

### clippy.toml

```toml
# clippy.toml (new file at repo root)
# Selective lints at warn level — catches killer bugs without -D-warnings churn.
# Use `#[expect(LINT_NAME, reason = "...")]` for justified exceptions (Rust 1.81+).

# Don't Panic — UTF-8 boundary + array/string indexing panics
string_slice            = "warn"
indexing_slicing        = "warn"

# Don't Panic — Result/Option unwraps and panic! macros
unwrap_used             = "warn"
panic                   = "warn"

# Skipped (not enabled in v0.3.35):
#   todo / unimplemented / unreachable — would flag LazyLock init sites; defer
#   get_unwrap / panic_in_result_fn — value-add marginal; defer
#   let_underscore_future / let_underscore_must_use — no current async bugs;
#     defer until tokio usage grows
#   cast_possible_truncation / cast_possible_wrap / cast_precision_loss —
#     no current numeric bugs; defer
#   arithmetic_side_effects — Evan Schwartz (Apr 2026) found 85% noise; defer
# Reference: https://emschwartz.me/your-clippy-config-should-be-stricter/
```

**Why `warn` not `deny`:** `#[expect(LINT, reason = "...")]` requires `warn`-level for the `expect` to be matched (Rust 1.81+). Using `deny` would force `#[allow]` only, losing the audit trail of "why is this allowed?".

**Known acceptances to add at implementation time:**
- `src/core/markdown.rs:42` — `Regex::new(...).unwrap()` inside `LazyLock::new(...)`. Justification: LazyLock initialization is one-shot at first access; a regex pattern that fails to compile is a developer bug we want caught at first use, not silently ignored. Wrap with `#[expect(unwrap_used, reason = "LazyLock init; pattern is hardcoded and tested")]`.
- Test files: ~10-15 `unwrap()` / `.expect()` calls in `tests/`. Wrap each with `#[expect(unwrap_used, reason = "test assertion")]` or use the existing `tempfile`/`predicates` patterns that avoid unwrap.

## Data flow (frontmatter parse path)

### Before

```
md file → parse_frontmatter → ParsedPage { frontmatter: serde_yaml::Value, body }
                                ↓
                         cli::ls/cli::lint/cli::embed/cli::status
                                ↓
                         value["type"].as_str().unwrap_or("page")
                         value["tags"].as_sequence().map(|s| s.iter().filter_map(|v| v.as_str()))
                         value["title"].as_str().unwrap_or("")
                                ↓
                         Runtime errors caught only by lint; typos silent.
```

### After

```
md file → parse_frontmatter → ParsedPage { frontmatter: Frontmatter, body }
                                ↓
                         cli::ls/cli::lint/cli::embed/cli::status
                                ↓
                         fm.page_type.as_deref().unwrap_or("page")
                         fm.tags.iter()
                         fm.title.as_deref().unwrap_or("")
                                ↓
                         Typos in known fields: serde-saphyr fails at parse.
                         Unknown fields preserved in fm.extra.
                         Schema for known fields in skills/references/frontmatter.schema.json.
```

## Error handling

- `serde_saphyr::Error` is `#[from]`-compatible, same as `serde_yaml::Error`. The `WikiError::Yaml` variant keeps its transparent display; only the source crate changes.
- Duplicate-key YAML is still an error (both crates reject it). Existing `lint_resilient_to_unparseable_frontmatter` test continues to pass unchanged.
- `Frontmatter` deserialization failures are caught by the `?` in `parse_frontmatter`; callers (`cli::ls` etc.) already handle unparseable pages via the existing `frontmatter-yaml-parse` lint code.
- New failure mode: a wiki page with **no recognized known fields but only niche `extra` fields** still parses successfully (extra captures them). No regression.

```rust
#[test]
fn parse_frontmatter_returns_typed_struct() {
    let md = "---\ntitle: Hello\ntags: [a, b]\n---\nBody";
    let p = parse_frontmatter(md).unwrap();
    assert_eq!(p.frontmatter.title.as_deref(), Some("Hello"));
    assert_eq!(p.frontmatter.tags, vec!["a", "b"]);
}

#[test]
fn parse_frontmatter_catches_known_field_typo() {
    // "titel" instead of "title" goes into `extra`, not into `title`
    let md = "---\ntitel: Hello\n---\n";
    let p = parse_frontmatter(md).unwrap();
    assert_eq!(p.frontmatter.title, None);
    assert_eq!(p.frontmatter.extra.get("titel").and_then(|v| v.as_str()), Some("Hello"));
}

#[test]
fn parse_frontmatter_preserves_niche_extra_fields() {
    // avatar is in only 1 of 4 real wikis; goes into extra
    let md = "---\ntitle: X\navatar: /img/x.png\ntimezone: Europe/Paris\n---\n";
    let p = parse_frontmatter(md).unwrap();
    assert_eq!(p.frontmatter.extra.len(), 2);
}

#[test]
fn parse_frontmatter_handles_duplicate_top_level_keys() {
    // Same as existing test — must still error
    let md = "---\ntype: page\ntype: query\n---\n";
    assert!(parse_frontmatter(md).is_err());
}

#[test]
fn parse_frontmatter_handles_empty_yaml() {
    let md = "---\n---\nBody";
    let p = parse_frontmatter(md).unwrap();
    assert_eq!(p.frontmatter.title, None);
    assert!(p.frontmatter.tags.is_empty());
}

#[test]
fn parse_frontmatter_handles_pipeline_type_expression() {
    // From minimax-code-wiki audit
    let md = "---\ntype: entity | concept | comparison | query | schema | summary\n---\n";
    let p = parse_frontmatter(md).unwrap();
    assert_eq!(p.frontmatter.page_type.as_deref(),
               Some("entity | concept | comparison | query | schema | summary"));
}

Smoke test against all 4 wikis. If a field is missing, the error message from serde-saphyr names the unknown field — add to struct, re-run. |
| serde-saphyr pinned at a version with a regression | Low (serde-saphyr is actively maintained; last update 2026-06-23 per the dep landscape research) | Medium | Fallback: `serde_yml = "0.0.13"` shim — same one-liner migration shape, explicitly endorsed by serde_yml's own docs as a stop-gap. |
| `clippy.toml` lints flag dozens of pre-existing sites | Medium (conservative 4-lint subset chosen to minimize this) | Medium | `#[expect(LINT, reason = "...")]` is the documented escape hatch. LazyLock init sites get a single `#[expect]` annotation with a real reason. Test files: ~10-15 expects, each with a 5-word reason. |
