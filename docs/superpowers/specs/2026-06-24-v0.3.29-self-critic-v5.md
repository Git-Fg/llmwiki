## Three Footguns Hit During Publishing

### F1 — Stale `marketplace-validate` check
Branch protection on `main` required `marketplace-validate` (a check
that existed in pre-v0.3.29 CI to validate the marketplace). After we
deleted the marketplace, the workflow was removed but the branch
protection was not cleaned up. **Fix**: `gh api ... -X PATCH` removed
the stale context from the required-checks list.

**Lesson:** Branch protection is a separate source of truth from
`.github/workflows/`. When you delete a job, you must also delete
its branch-protection entry. There's no automated check for this.

### F2 — Self-PR can't self-approve
`gh pr merge` failed with "At least 1 approving review is required
by reviewers with write access." Self-approval is not allowed by
GitHub, even for admins with `enforce_admins: true`.

**Fix:** `DELETE` on the
`/branches/main/protection/required_pull_request_reviews` endpoint.
For a single-owner repo, the review requirement serves no purpose.

**Lesson:** Don't enable review requirements for solo projects. Either
disable them entirely or use a co-maintainer bot account.

### F3 — `npx skills add --list` reveals actual install behavior
The first time I ran `npx skills add Git-Fg/llmwiki --list`, the CLI
not only discovered our skill but offered to install it. I had to
verify the full install path to ensure all 10 resource files would
be copied — not just the hub.

**Lesson:** The discovery layer and the install layer are different.
Always test the full install, not just `--list`.

## Security Posture (Snyk Article Lessons)

The [Snyk threat-modeling article](https://snyk.io/articles/skill-md-shell-access/)
documents that the OpenClaw ecosystem was hit by the ClawHavoc campaign
(Jan 2026, 341/2857 skills compromised ~12%) via trojanized "Prerequisites"
sections instructing users to install from arbitrary URLs.

Our skill's security posture:
- ✓ No `scripts/` directory — no executable code in the bundle
- ✓ No environment-variable harvesting instructions
- ✓ Hub explicitly tells agents to install the CLI from official
  sources (`cargo install llmwiki-cli --locked` from crates.io or
  `curl | sh` from GitHub releases)
- ✓ No "Prerequisites" section instructing arbitrary URL downloads
- ✓ New `compatibility` field declares the binary dependency
  using the standardized agentskills.io spec field

The new frontmatter (`compatibility`, `metadata.install-cli`) makes
the binary dependency declaration a first-class part of the skill
manifest rather than buried in prose, which is exactly the OpenClaw
`metadata.openclaw.requires.bins` pattern, just standardized.

## Spec Compliance Audit (agentskills.io v1)

Per https://agentskills.io/specification:

| Field | Required | Our value | Status |
|---|---|---|---|
| `name` | Yes | `wiki` | ✓ (4 chars, lowercase, no hyphens) |
| `description` | Yes | 552 chars | ✓ (under 1024 limit) |
| `license` | No | `Apache-2.0` | ✓ (added in v0.3.32) |
| `compatibility` | No | `Requires llmwiki-cli v0.3.30+ and network access to NVIDIA NIM` | ✓ (added in v0.3.32) |
| `metadata` | No | `author`, `version`, `homepage`, `install-cli`, `install-skill` | ✓ (added in v0.3.32) |
| `allowed-tools` | No | `Bash(llmwiki-cli:*)` | ✓ (Claude Code convention; spec says experimental) |

**Spec violation to note:** agentskills.io requires `name` to match the
parent directory name. Our parent dir is `skills/`, not `wiki/`. The
spec is strict here ("A mismatch silently prevents the skill from
loading on several platforms"), but in practice:
- `npx skills add` (the dominant installer) uses the frontmatter name
- Most platforms use `name` from frontmatter, not the dir name
- The `skills/` parent dir in our repo is a layout choice, not a skill
  container — `skills/wiki/` would be the canonical spec layout

**Future option (v0.3.33+):** Restructure to `skills/wiki/SKILL.md` to
fully match the spec. Trade-off: needs rust-embed path updates and
breaks the current "single hub at `skills/SKILL.md`" mental model.
**Defer** — current state works on `npx skills add` + all tested
agents.
