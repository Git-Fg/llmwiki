## Key Research Findings

### F1 — `npx skills add` compatibility (BLOCKING → RESOLVED)

**Source:** `vercel-labs/skills` source code (`src/blob.ts`), 23k stars,
72+ agents supported.

The `findSkillMdPaths()` function in `blob.ts` has a `PRIORITY_PREFIXES`
list that includes `'skills/'`. Discovery handles three depths:

1. `skills/SKILL.md` — direct (our layout, discovered as skill "wiki")
2. `skills/<name>/SKILL.md` — one level deep (standard agentskills.io)
3. `skills/<category>/<name>/SKILL.md` — bounded depth-2 (new)

**Conclusion:** Our `skills/SKILL.md` layout IS discovered as a single
skill named "wiki" (from frontmatter `name: wiki`). All sibling `.md`
files are bundled as resources. **No restructuring needed.** The repo is
already `npx skills add felix/llmwiki` compatible (once published to GitHub).

**Tension resolved:** H1 (spec §2.1 Goal #1 "10 entries on disk" vs
implementation "1 entry on disk") was a false dichotomy.
- `llmwiki-cli install-skill` = small install (hub only, 1 file, runtime
  sub-skill serving via `skill get`). Correct for CLI users.
- `npx skills add` = fat install (all files, 10 entries on disk). Correct
  for non-CLI users.
- Both are valid distribution channels for the same repo layout.

### F3 — agentskills.io canonical directory layout

```
my-skill/
├── SKILL.md          # Required
├── scripts/          # Optional
├── references/       # Optional

```

**Our compliance:**

- Sub-skills as flat `.md` files ✓ (resource files within the skill)
- Missing: `scripts/`, `assets/` — not needed currently

Cursor added SKILL.md support alongside its legacy `.cursorrules` system.

**Status:** Not a code issue. When the repo is pushed to GitHub as (e.g.)
`felix/llmwiki`, users will be able to run `npx skills add felix/llmwiki`
to install the full skill bundle. No additional repo config needed — the

### M1 — Symlink vs copy (informational)

`npx skills add` defaults to symlink (single source of truth, easy
updates). Our `install-skill` copies. This is a deliberate choice: the
CLI binary embeds the skill bytes via rust-embed, so the "source of
truth" is the binary itself, not a symlink target. Copying is correct
for our model. No action needed.
